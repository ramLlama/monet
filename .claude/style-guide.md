# Monet Style Guide

## Language & Conventions

This is an Emacs Lisp package following standard Elisp conventions with `lexical-binding: t`.

## Naming

- **Package prefix**: `monet-` for public symbols, `monet--` for private/internal symbols
- **Struct fields**: accessed via `monet--session-<field>` (auto-generated by `cl-defstruct`)
- **Predicates**: use `-p` suffix (e.g., `monet--port-available-p`, `monet--should-show-diff-p`, `monet--is-buffer-relevant-to-session-p`)
- **Customization groups**: `monet` (general), `monet-tool` (MCP tools), `monet-ediff` (ediff settings)
- **Tool handler functions**: `monet--tool-<name>-handler` for MCP protocol adapters
- **Default implementations**: `monet-default-<name>-tool` for replaceable tool functions
- **Constants**: `monet--<name>` with no special casing (e.g., `monet--port-min`, `monet--protocol-version`)

## File Organization (within monet.el)

The file is organized in this order:
1. Header / Package-Requires
2. Dependencies (`require` forms)
3. Customization groups and `defcustom` declarations
4. Constants
5. Data structures (`cl-defstruct`)
6. Utility functions
7. Session management
8. WebSocket communication (sending, receiving, server lifecycle)
9. MCP protocol handlers (initialize, resources, tools)
10. Tool implementations (default implementations for each MCP tool)
11. MCP tool list definition
12. Resource definitions
13. Hooks registration
14. Diff visibility management
15. Selection tracking
16. Diff tools (simple diff, ediff, cleanup)
17. Additional tool implementations (open file, save, diagnostics, etc.)
18. Logging
19. User commands (start/stop/list servers)
20. Minor mode definition
21. `(provide 'monet)`

## Docstrings

- Every `defun`, `defvar`, `defcustom`, and `defconst` has a docstring
- First line is a complete sentence (imperative mood for functions)
- Parameters are documented in ALLCAPS in the docstring body
- The `checkdoc` Makefile target enforces docstring conventions (with `sentence-end-double-space` set to nil)

## JSON Handling

- Encoding: `json-encode` with alist data structures (cons cells)
- Decoding: `json-read-from-string`
- Boolean values: `t` for true, `:json-false` for false (Emacs json.el convention)
- Empty arrays: `[]` (symbol) or `(vector)` / `(vconcat list)`
- Null: `nil` (but watch for alist vs plist confusion)

## Error Handling

- Use `condition-case` for recoverable errors
- Error codes follow JSON-RPC 2.0: `-32601` (method not found), `-32602` (invalid params), `-32603` (internal error)
- Log errors via `message` (appears in *Messages* buffer)
- Lockfile/server errors use `error` to signal

## Patterns

### Alist construction with backquote
```elisp
`((key1 . ,value1)
  (key2 . ,value2)
  ,@(when condition `((optional-key . ,optional-value))))
```

### Deferred work via timers
- `run-with-timer` for delayed execution
- `run-with-idle-timer` for work after user is idle
- Common pattern for post-action cleanup (e.g., 200ms delay after diff reject)

### Debouncing via `post-command-hook`
- Cancel existing timer, set new one
- Used for selection tracking (50ms) and diff visibility (100ms)

### Tool customization pattern
```elisp
(defcustom monet-<tool>-tool 'monet-default-<tool>-tool ...)
;; Handler extracts params, calls (funcall monet-<tool>-tool ...)
;; Default implementation does the actual work
```

### MCP response format
All tool results are lists of content objects:
```elisp
(list `((type . "text")
        (text . ,json-string-or-status)))
```

## Anti-patterns to Avoid

- Do not use `with-current-buffer` without checking `buffer-live-p` first when the buffer may have been killed
- Do not send MCP responses for notifications (methods without `id`)
- Do not assume `buffer-file-name` is non-nil -- always guard with `when`
- Do not block the Emacs event loop in tool handlers -- use deferred responses for interactive flows
